## 002: Plausibility-gate the rebuild — replace only when provably wrong (2026-04-30)

**Decision:** `rebuild-migration-bib.py` does not blindly replace migration entries with WP-derived ones. It first runs a plausibility check on the existing entry (cite-key surname must appear in author/editor; title fragment must overlap entry title or WP title; reject all-caps titles and `"Place of publication not identified"`). Plausible entries are kept as-is, preserving any later enrichment (URL, ISBN, file path, edition, publisher). Only entries that fail the check get rebuilt from WP.

**Rejected:**
- **Replace every WP-covered entry unconditionally**: First pass did this and produced 580 rebuilds, many of which clobbered correct, well-enriched entries (Cicero's Clarendon edition, Wittgenstein's Pears/McGuinness translation, etc.) with bare title-only stubs. Discovered when reviewing the first side-by-side report.
- **Heuristic-only "looks like junk" detection** (regex for `EISENBRAUNS`, all-caps publisher, etc.): Catches the obvious cases but misses entries like `Alexander2014SlateStarCodex` that have full ISBN/DOI/publisher metadata for a *completely unrelated work* (Indonesian pneumatic-drive conference paper). Need the cite-key-vs-content cross-check.
- **Trust the cite key surname even with no other signal**: For cite keys without years (`CiceroAmicitia`, `MontaigneEssais`), the regex initially treated the whole string as the surname — false-positive mismatch with author "Cicero, Marcus Tullius". Required a separate code path: with year → before digits; no year → first CamelCase chunk.

**Files:** `scripts/rebuild-migration-bib.py` — `migration_entry_is_plausible()`, `cite_key_surname()`, `cite_key_title_words()`.
